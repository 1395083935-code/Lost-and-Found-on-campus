## 🎯 应急响应概述

### 响应目标
- **P0 (严重故障)**: 30 分钟内恢复
- **P1 (高优先)**: 2 小时内恢复
- **P2 (中等优先)**: 1 个工作日内恢复
- **P3 (低优先)**: 2-3 个工作日内恢复

### 响应原则
1. **快速定位**: 尽快确定故障原因
2. **优先恢复**: 先恢复服务，后分析细节
3. **减少影响**: 最小化用户受影响范围
4. **全面记录**: 记录所有排查步骤和结果
5. **事后改进**: 从每次故障中学习并改进系统

## 🚨 故障分级

### P0 级别 (严重故障) - 24h 应急响应

**症状**:
- 整个系统无法访问 (无法打开首页)
- 数据库完全不可用 (无法查询)
- 所有 API 返回 500 错误 (> 10% 错误率)
- 造成生产数据丢失

**例子**:
- 后端 Java 进程完全宕机
- 数据库磁盘满，无法写入
- 数据库连接池耗尽
- 整个数据库崩溃

**应急资源**:
- 全体技术人员 24h 待命
- 可实施所有紧急措施 (包括版本回滚)

**目标恢复时间**: **30 分钟**

### P1 级别 (高优先) - 工作日立即响应

**症状**:
- 核心功能无法使用 (如搜索、发布)
- 部分用户无法登录 (> 1% 用户受影响)
- 数据不一致 (丢失或重复)
- API 响应时间 > 5 秒

**例子**:
- 搜索功能崩溃
- 数据库主从同步失败
- 内存泄漏导致 OOM
- 某个数据表损坏

**应急资源**:
- 值班工程师立即介入
- 可实施重启、回滚等措施

**目标恢复时间**: **2 小时**

### P2 级别 (中等优先) - 工作日 4 小时内响应

**症状**:
- 部分功能缓慢 (但可用)
- 个别用户反馈故障 (< 1% 用户)
- 性能下降 (P95 > 500ms)
- 非关键数据不一致

**例子**:
- 物品列表加载缓慢
- 热搜词查询超时
- 某个 API 偶发错误
- 日志文件过大

**应急资源**:
- 值班工程师值班期间响应
- 可实施性能优化、清理缓存等措施

**目标恢复时间**: **1 个工作日**

### P3 级别 (低优先) - 按计划处理

**症状**:
- 非核心功能故障
- 易出现的问题，不影响使用
- 文档、页面显示问题

**例子**:
- 页面加载很慢但最终能打开
- 帮助文档错别字
- UI 显示不完全
- 缓存数据过期

**应急资源**:
- 集成到下一周期修复

**目标处理时间**: **2-3 个工作日**

## 🔄 故障响应流程

### 故障发现 (T+0 秒)

```
用户反馈 / 监控告警
    ↓
确认故障 (重现问题)
    ↓
快速分类 (P0/P1/P2/P3)
    ↓
启动应急流程
```

**快速确认检查清单**:
```bash
# 1. 检查后端服务
curl -I http://localhost:8082/actuator/health

# 2. 检查前端服务  
curl -I http://localhost:8081/

# 3. 检查数据库
mysql -u root -p -e "SELECT 1;"

# 4. 检查系统资源
top | head -10
df -h
```

### P0 故障应急流程 (< 30 分钟)

```
T+0:   故障发现 & 快速分类
       → 启动紧急通知 (Slack/钉钉/微信)
       → 技术负责人确认故障
       
T+5:   快速诊断 (从 3 个维度)
       → 检查后端进程 & 日志
       → 检查数据库连接 & 磁盘
       → 检查前端服务 & 网络
       
T+10:  实施紧急恢复 (按优先级)
       → 尝试 1: 重启服务 (重启 Java 进程)
       → 尝试 2: 故障转移 (如有备份实例)
       → 尝试 3: 数据库恢复 (从备份恢复)
       → 尝试 4: 版本回滚 (回到上一版本)
       
T+20:  验证服务恢复
       → 健康检查: /actuator/health
       → 业务验证: 测试核心功能
       → 数据验证: 检查数据一致性
       
T+30:  恢复确认 & 通知
       → 确认系统恢复正常
       → 通知所有用户服务已恢复
       → 开始记录详细日志供事后分析
```

### P1 故障应急流程 (< 2 小时)

```
T+0-5:   故障分类 & 初步诊断
         → 收集用户反馈
         → 查看错误日志
         → 确定故障范围
         
T+5-30:  深入排查 & 制定方案
         → 定位具体模块 (后端/前端/数据库)
         → 收集相关日志和指标
         → 评估修复方案优先级
         
T+30-90: 执行修复
         → 根据故障类型选择修复方法
         → 可能操作: 重启、配置调整、SQL 修复等
         → 持续监控修复效果
         
T+90-120: 验证 & 总结
         → 功能完整性测试
         → 性能指标确认
         → 发送恢复通知
         → 初步分析根本原因
```

### P2 故障应急流程 (< 1 个工作日)

```
T+0:     故障登记
         → 建立 Issue 追踪
         → 指派责任工程师
         
T+4h:    排查与分析
         → 分析日志
         → 重现问题
         → 确定根本原因
         
T+8h:    修复与验证
         → 编写修复代码
         → 测试修复效果
         → 提交代码审查
         
T+24h:   部署与发布
         → 完成 code review
         → 部署到生产环境
         → 验证问题已解决
```

## 🔧 常见故障处理

### 故障 1: 后端无法启动 (P0)

**症状**:
```
后端进程无法启动，或启动后立即退出
curl: (7) Failed to connect to localhost port 8082
```

**快速诊断**:
```bash
# 查看启动日志 (最后 50 行)
tail -50 logs/app.log | grep "ERROR\|Exception"

# 检查端口占用
lsof -i :8082

# 检查磁盘空间
df -h /

# 检查数据库连接
mysql -u campus_user -p -e "SELECT 1;"
```

**常见原因及解决方案**:
| 原因 | 症状 | 解决方案 |
|------|------|--------|
| 端口被占用 | "Address already in use" | `lsof -i :8082` 并 kill 占用进程 |
| 数据库不可用 | "Connection refused" | 检查 MySQL 是否运行，连接参数是否正确 |
| 磁盘满 | "No space left on device" | 清理日志、备份，扩展磁盘 |
| 内存不足 | OOM killer 日志 | 增加堆内存 `-Xmx1024m` |
| 配置错误 | "Property 'xxx' not found" | 检查 application.yml 配置文件 |

**恢复步骤**:
```bash
# 1. 停止任何现有进程
pkill -f "campus-backend"
sleep 5

# 2. 查看日志定位问题
tail -100 logs/app.log

# 3. 修复问题 (根据具体原因)

# 4. 重新启动
nohup java -Xms512m -Xmx1024m -jar campus-backend-0.0.1-SNAPSHOT.jar \
  --spring.profiles.active=prod > logs/app.log 2>&1 &

# 5. 验证启动成功
sleep 10
curl http://localhost:8082/actuator/health
```

### 故障 2: 前端页面白屏 (P1)

**症状**:
```
打开首页显示空白，浏览器控制台有 JavaScript 错误
Network 标签显示 API 返回 500 或无响应
```

**快速诊断**:
```bash
# 1. 检查后端服务
curl http://localhost:8082/api/items

# 2. 检查前端服务
curl http://localhost:8081/ | head -50

# 3. 查看浏览器控制台 (F12)
# - Network 标签: 查看具体 API 返回
# - Console 标签: 查看 JavaScript 错误
```

**常见原因及解决方案**:
| 原因 | 症状 | 解决方案 |
|------|------|--------|
| 后端不可用 | API 返回 Connection refused | 重启后端服务 |
| 跨域问题 | CORS error | 检查后端 CORS 配置 |
| 前端资源加载失败 | 404 错误 | 重新构建部署前端 |
| JavaScript 错误 | 控制台红色错误 | 检查浏览器版本兼容性 |

**恢复步骤**:
```bash
# 1. 验证后端可用
curl http://localhost:8082/api/items

# 2. 如果后端不可用，重启
cd /opt/campus-backend
pkill -f java
nohup java -jar campus-backend-0.0.1-SNAPSHOT.jar ... &

# 3. 清理浏览器缓存 (Ctrl+Shift+Delete)

# 4. 如果问题持续，重新部署前端
npm run build
scp -r dist/* user@server:/usr/share/nginx/html/campus/

# 5. 重启 Nginx
systemctl restart nginx

# 6. 验证
curl http://localhost:8081/
```

### 故障 3: 数据库查询超时 (P1)

**症状**:
```
API 响应超时: "Query timeout after 30s"
数据库连接数迅速增长
服务器 CPU 突增
```

**快速诊断**:
```bash
# 1. 查看数据库状态
mysql -u root -p -e "SHOW PROCESSLIST;" | grep -i sleep

# 2. 查看慢查询日志 (需要提前启用)
tail -50 /var/log/mysql/slow.log

# 3. 检查是否有锁定的表
mysql -u root -p -e "SHOW OPEN TABLES WHERE In_use > 0;"

# 4. 查看数据库磁盘使用
du -sh /var/lib/mysql/campus_db
```

**常见原因及解决方案**:
| 原因 | 症状 | 解决方案 |
|------|------|--------|
| 缺少索引 | 全表扫描 | 创建缺失的索引 |
| 查询条件不当 | WHERE 子句效率低 | 重写 SQL 查询 |
| 表过大 | 1000+ 万条记录 | 分表分库、归档旧数据 |
| 连接数耗尽 | "Too many connections" | 增加连接池大小 |

**恢复步骤**:
```sql
-- 1. 杀死长时间运行的查询
KILL QUERY 进程ID;

-- 2. 查看具体慢查询
EXPLAIN SELECT * FROM lost_found_item 
WHERE status = 1 AND create_time > '2026-04-01' 
ORDER BY create_time DESC LIMIT 100;

-- 3. 如果缺少索引，创建
CREATE INDEX idx_status_create_time 
ON lost_found_item(status, create_time);

-- 4. 刷新缓存
FLUSH QUERY CACHE;

-- 5. 重启连接池
-- (需要重启后端应用)
```

### 故障 4: 内存溢出 (OOM) (P1)

**症状**:
```
后端进程被 kill: "Killed (OOM)"
应用日志: "java.lang.OutOfMemoryError"
系统日志: "Memory cgroup out of memory"
```

**快速诊断**:
```bash
# 1. 查看进程内存使用
ps aux | grep java | grep campus

# 2. 查看 GC 日志
grep "GC\|OutOfMemory" logs/app.log

# 3. 查看系统内存
free -h

# 4. 查看最大堆内存设置
grep "Xmx" startup.sh
```

**常见原因及解决方案**:
| 原因 | 症状 | 解决方案 |
|------|------|--------|
| 堆内存设置过小 | 频繁 GC | 增加 -Xmx 参数 |
| 内存泄漏 | 堆内存持续增长 | 分析 dump 文件，修复泄漏 |
| 缓存未清理 | 缓存对象堆积 | 清理过期缓存或降低缓存大小 |
| 连接未释放 | 连接对象堆积 | 检查数据库连接是否正确关闭 |

**恢复步骤**:
```bash
# 1. 停止应用并增加堆内存
pkill -f "campus-backend"

# 2. 修改启动脚本，增加 -Xmx
# 将 -Xmx1024m 改为 -Xmx2048m

# 3. 重新启动
nohup java -Xms512m -Xmx2048m -jar campus-backend-0.0.1-SNAPSHOT.jar \
  --spring.profiles.active=prod > logs/app.log 2>&1 &

# 4. 监控内存使用
watch -n 5 'ps aux | grep java'

# 5. 如果仍然频繁 OOM，可能是内存泄漏，需要
# 生成堆 dump 文件供分析
jmap -dump:live,format=b,file=heap.bin <PID>
```

### 故障 5: 版本回滚 (紧急情况)

**何时使用**:
- 新版本引入严重 bug 或安全漏洞
- 数据库迁移失败导致数据损坏
- 完全无法快速定位和修复问题

**回滚步骤** (< 15 分钟):
```bash
# 1. 确认前一版本 JAR 位置
ls -la /opt/campus-backend/*.jar

# 2. 停止当前应用
pkill -f "campus-backend"
sleep 5

# 3. 恢复数据库 (如果有数据库变更)
mysql -u root -p campus_db < /var/backups/campus_db_YYYYMMDD_HHMMSS.sql

# 4. 启动前一版本
cd /opt/campus-backend
nohup java -jar campus-backend-0.0.1-SNAPSHOT-old.jar \
  --spring.profiles.active=prod > logs/app.log 2>&1 &

# 5. 验证回滚成功
sleep 10
curl http://localhost:8082/actuator/health

# 6. 通知团队
# 发送通知: "系统已回滚到上一版本，故障正在调查"

# 7. 启动 Post-mortem 流程
# 分析为什么新版本会有此问题
# 制定改进措施避免再次发生
```

## 📋 事后处理 (Post-Mortem)

### Post-Mortem 会议 (故障后 24-48 小时)

#### 参加人员
- 故障响应负责人
- 技术负责人
- 相关工程师 (后端、前端、数据库)
- 运维负责人
- 产品负责人 (了解业务影响)

#### 会议议程 (1 小时)
1. **故障回顾** (10 分钟)
   - 故障时间、持续时间、影响范围
   - 用户投诉情况

2. **根本原因分析** (15 分钟)
   - 故障的直接原因是什么?
   - 为什么没有提前发现?
   - 为什么恢复花了那么长时间?

3. **改进措施** (20 分钟)
   - 技术改进: 代码、架构、监控
   - 流程改进: 部署、测试、告警
   - 文档更新: 故障案例、排查手册

4. **后续行动** (15 分钟)
   - 谁负责什么改进项目?
   - 截止时间是什么时候?
   - 如何验证改进有效?

#### Post-Mortem 模板
```markdown

- 影响用户数:

- 直接原因:
- 根本原因:

## Corrective Actions
- 短期 (立即): [措施] - [负责人] - [截止日期]
- 中期 (1 个月): [措施] - [负责人] - [截止日期]
- 长期 (3 个月): [措施] - [负责人] - [截止日期]
```

### 改进示例
```

改进措施:
1. 技术改进
   - 添加数据库主从同步监控告警

   - 每周检查主从同步状态

4. 测试改进
   - 增加主从同步异常的集成测试
   - 演练数据库故障场景
```
